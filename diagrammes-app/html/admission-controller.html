Points clés — flux

  • Le webhook n'intercepte que CREATE et UPDATE — pas DELETE
  • L'Admission Controller consulte le Sensor local pour évaluer les policies
  • Sensor initie la connexion vers Central — pas l'inverse
  • Si l'Admission Controller est indisponible et failOpen: true → requêtes autorisées par défaut

Configuration dans la CR SecuredCluster

apiVersion: platform.stackrox.io/v1alpha1
kind: SecuredCluster
metadata:
  name: stackrox-secured-cluster-services
  namespace: stackrox
spec:
  clusterName: mon-cluster
  centralEndpoint: central.stackrox.svc:443
  admissionControl:
    listenOnCreates: true   # intercepte CREATE
    listenOnUpdates: true   # intercepte UPDATE
    listenOnEvents: true    # événements Kubernetes
    enforceOnCreates: true # hard enforcement sur CREATE
    enforceOnUpdates: true # hard enforcement sur UPDATE
    bypass: Disabled       # désactive le bypass annotation
    replicas: 3          # HA — 3 replicas recommandés
  sensor:
    resources: # optionnel
  collector:
    collection: EBPF         # ou KERNEL_MODULE

Paramètres clés de admissionControl

bypass
Disabled — aucun bypass possible (recommandé prod)
BreakGlassAnnotation — bypass via annotation sur le pod
Annotation : admission.stackrox.io/break-glass: "true"
listenOnCreates
true — intercepte les CREATE
false — ignore les créations (déconseillé)
Par défaut : true
listenOnUpdates
true — intercepte les UPDATE
Important pour détecter les modifications post-déploiement
Par défaut : true
enforceOnCreates
true — Hard enforcement sur CREATE (bloc API)
false — Inform only sur CREATE
Nécessite listenOnCreates: true
enforceOnUpdates
true — Hard enforcement sur UPDATE
false — Inform only sur UPDATE
Nécessite listenOnUpdates: true
replicas
3 — recommandé pour la HA
1 — acceptable en dev/test
Si tous les pods tombent → failOpen détermine le comportement

Dépannage de l'Admission Controller

Tous les déploiements sont bloqués
Policy trop restrictive ou mal configurée bloquant tout le cluster
Solution d'urgence — supprimer le webhook : oc delete validatingwebhookconfiguration stackrox Puis investiguer et corriger la policy concernée.
Policy enforce configurée mais ne bloque pas
Admission Controller non déployé ou bypass activé
Vérifier :
oc get pods -n stackrox -l app=admission-control
oc get validatingwebhookconfigurations | grep stackrox
Timeout lors des déploiements
Pods admission-control surchargés ou réseau lent vers Sensor
Vérifier les logs : oc logs -n stackrox -l app=admission-control --tail=50 Augmenter le nombre de replicas si nécessaire.
Besoin de bypass d'urgence sur un pod spécifique
Policy bloque un workload critique en production
Si bypass: BreakGlassAnnotation configuré : metadata:
  annotations:
    admission.stackrox.io/break-glass: "true"
Admission Controller non créé après CR SecuredCluster
Init bundle manquant ou secrets non appliqués dans le namespace stackrox
Vérifier les secrets : oc get secrets -n stackrox | grep admission Réappliquer le bundle si nécessaire.
ValidatingWebhookConfiguration absent
CR SecuredCluster non déployée ou listenOnCreates: false
Vérifier la CR : oc get securedcluster -n stackrox -o yaml | grep admission Vérifier que listenOnCreates: true et enforceOnCreates: true.

Commandes de diagnostic clés

  • Lister les pods : oc get pods -n stackrox -l app=admission-control
  • Logs admission controller : oc logs -n stackrox -l app=admission-control
  • Vérifier le webhook : oc get validatingwebhookconfiguration | grep stackrox
  • Supprimer le webhook (urgence) : oc delete validatingwebhookconfiguration stackrox
  • Vérifier la CR SecuredCluster : oc get securedcluster -n stackrox -o yaml

Résumé pour l'examen

  • Webhook = ValidatingWebhookConfiguration (Validating, pas Mutating)
  • Intercepte CREATE et UPDATE uniquement — pas DELETE
  • 3 replicas recommandés pour la HA
  • bypass: Disabled = aucun contournement possible (recommandé)
  • bypass: BreakGlassAnnotation = bypass via annotation sur le pod
  • Déblocage d'urgence : oc delete validatingwebhookconfiguration stackrox
  • Hard enforcement → API server bloque (403) — pod jamais créé
  • Soft enforcement → pod créé, puis scale to 0 par Sensor
