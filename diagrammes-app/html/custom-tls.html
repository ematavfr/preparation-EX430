Certificats TLS Custom — RHACS Central

Objectif 1.3 — Configurer Central avec un certificat TLS personnalisé

🏠

Workflow — Configurer un certificat TLS custom pour Central

Par défaut, Central génère un certificat auto-signé. Ce workflow permet d'utiliser un cert signé par votre CA d'entreprise ou Let's Encrypt.
1
Obtenir / générer le certificat
CA d'entreprise · Let's Encrypt · openssl self-signed pour tests
2
Créer le Secret TLS dans OpenShift
oc create secret tls central-tls-custom --cert= --key= -n stackrox
3
Référencer le Secret dans la CR Central
spec.central.defaultTLSSecret.name: central-tls-custom
4
Appliquer et attendre le redémarrage automatique
oc apply -f central.yaml → Central redémarre → oc rollout status
5
Vérifier + gérer l'impact sur les Sensors
openssl s_client · Platform Configuration → Clusters · restart Sensor si besoin

CR Central — champ clé

apiVersion: platform.stackrox.io/v1alpha1
kind: Central
metadata:
  name: stackrox-central-services
  namespace: stackrox
spec:
  central:
    defaultTLSSecret:
      name: central-tls-custom   # ← nom du Secret kubernetes.io/tls dans stackrox
    exposure:
      route:
        enabled: true
    persistence:
      persistentVolumeClaim:
        storageClassName: gp3
⚠️ Le Secret doit être dans le namespace stackrox (même namespace que Central) et de type kubernetes.io/tls.

Types de certificats dans RHACS

Certificat TLS Central (UI/API)

  • Présenté aux navigateurs et à roxctl
  • Configurable via defaultTLSSecret
  • Type : kubernetes.io/tls
  • Contenu : tls.crt + tls.key

Certificats internes (mTLS)

  • Communication Central ↔ Sensor ↔ Collector
  • Gérés automatiquement par RHACS (auto-signés internes)
  • Distribués via l'init bundle
  • Non remplaçables par un cert custom utilisateur

Init Bundle (secrets TLS composants)

  • collector-tls
  • sensor-tls
  • admission-control-tls
  • Expiration : 1 an (renouvelable)

Certificats Route OpenShift

  • La Route peut avoir son propre TLS (edge termination)
  • Indépendant de defaultTLSSecret
  • Configurable dans la Route elle-même
  • Cas d'usage : wildcard cert du cluster OCP
CertQui le configureMécanismeRenouvellement
TLS Central UI/API Administrateur RHACS spec.central.defaultTLSSecret Renouveler le Secret TLS + redémarrage Central
mTLS interne RHACS automatiquement Géré par Central CA interne Automatique (via init bundle renew)
Init bundle Admin RHACS (roxctl) roxctl central init-bundles generate Générer nouveau bundle + appliquer + restart Sensor

Commandes — Étape par étape

Étape 1 — Générer un certificat (si CA interne avec openssl)

# Générer une clé privée
openssl genrsa -out central.key 2048

# CSR (Certificate Signing Request)
CENTRAL_HOST=$(oc get route central -n stackrox -o jsonpath='{.spec.host}')
openssl req -new -key central.key -out central.csr \
  -subj "/CN=$CENTRAL_HOST/O=RHACS"

# Signer avec votre CA (exemple)
openssl x509 -req -in central.csr \
  -CA ca.crt -CAkey ca.key -CAcreateserial \
  -out central.crt -days 365

# Vérifier la chaîne : le cert doit inclure les intermédiaires
cat central.crt ca.crt > central-full-chain.crt

Étape 2 — Créer le Secret TLS dans OpenShift

# Créer le Secret (type kubernetes.io/tls automatiquement)
oc create secret tls central-tls-custom \
  --cert=central-full-chain.crt \
  --key=central.key \
  -n stackrox

# Vérifier le Secret créé
oc get secret central-tls-custom -n stackrox -o yaml | grep "tls\."

Étape 3 — Mettre à jour la CR Central

# Patch en ligne (sans modifier le fichier YAML)
oc patch central stackrox-central-services -n stackrox \
  --type=merge \
  -p '{"spec":{"central":{"defaultTLSSecret":{"name":"central-tls-custom"}}}}'

# Ou via le fichier YAML complet :
oc apply -f central.yaml

Étape 4 — Suivre le redémarrage

# Central redémarre automatiquement après modification de la CR
oc rollout status deployment/central -n stackrox
# Waiting for deployment "central" rollout to finish: 0 of 1 updated replicas are available...
# deployment "central" successfully rolled out

Étape 5 — Vérifier le certificat présenté

CENTRAL_HOST=$(oc get route central -n stackrox -o jsonpath='{.spec.host}')

# Vérifier le cert présenté (sujet + dates)
echo | openssl s_client \
  -connect $CENTRAL_HOST:443 \
  -servername $CENTRAL_HOST \
  2>/dev/null | openssl x509 -noout -subject -issuer -dates

# Vérifier via curl (doit répondre sans -k si le CA est connu)
curl -v --cacert ca.crt https://$CENTRAL_HOST/v1/ping

Impact sur les Sensors après changement de certificat

⚠️ Risque : perte de connexion Sensor → Central

Le certificat TLS Custom ne change que le cert présenté aux navigateurs/roxctl.
La communication interne Central ↔ Sensor utilise le mTLS interne (init bundle) — non affecté.

Cependant : si le CA de confiance côté Sensor ne reconnaît pas le nouveau cert (cas du cert auto-signé custom sans CA distribué), des warnings peuvent apparaître.

✅ Dans la pratique — comportement attendu

Les Sensors ne perdent pas leur connexion lors du changement de defaultTLSSecret — car ils communiquent via le canal mTLS interne (port gRPC) qui utilise les certs de l'init bundle, indépendant du cert TLS de l'interface web.

Quand les Sensors perdent vraiment leur connexion

CauseSymptômeAction
Init bundle expiré (1 an) Cluster Unhealthy · logs : certificate has expired Générer nouveau bundle + oc apply + restart Sensor
CA interne RHACS régénéré (rare) Tous les Sensors perdent confiance simultanément Nouveau init bundle pour tous les clusters sécurisés
Secret init bundle supprimé par erreur Sensor CrashLoop · secret not found Re-appliquer le fichier YAML de l'init bundle

Vérifier l'état des Sensors après changement

# UI : Platform Configuration → Clusters → vérifier statut "Healthy"

# CLI
oc get securedcluster -n stackrox -o yaml | grep -A 5 conditions

# Si un Sensor est en erreur — restart
oc rollout restart deployment/sensor -n stackrox
oc rollout status deployment/sensor -n stackrox

Vérification du certificat

Checklist post-déploiement : cert valide → chaîne complète → dates cohérentes → Sensors toujours Healthy.

Checklist de vérification

✓ Cert présenté — openssl s_client -connect <host>:443 → CN correct
✓ Émetteur — openssl x509 -issuer → CA attendu (pas l'ancien auto-signé RHACS)
✓ Dates — openssl x509 -dates → notBefore / notAfter cohérents
✓ Chaîne complète — openssl verify -CAfile ca.crt central.crt → OK
✓ Sensors — Platform Configuration → Clusters → tous Healthy
✓ roxctl — roxctl central whoami -e $ROX_ENDPOINT → répond sans erreur TLS

Commandes de vérification complètes

CENTRAL_HOST=$(oc get route central -n stackrox -o jsonpath='{.spec.host}')

# 1. Vérifier le cert présenté
echo | openssl s_client \
  -connect $CENTRAL_HOST:443 \
  -servername $CENTRAL_HOST 2>/dev/null \
  | openssl x509 -noout -subject -issuer -dates

# 2. Vérifier la chaîne de confiance
openssl verify -CAfile ca.crt central.crt
# central.crt: OK

# 3. Tester l'API Central avec CA connu
curl --cacert ca.crt \
  https://$CENTRAL_HOST/v1/ping \
  -H "Authorization: Bearer $ROX_API_TOKEN"
# {"status":"ok"}

# 4. Vérifier via roxctl
roxctl central whoami -e "https://$CENTRAL_HOST:443"

# 5. État des clusters sécurisés
oc get securedcluster -n stackrox

Résumé pour l'examen

• Secret type kubernetes.io/tls dans le namespace stackrox
• CR Central : spec.central.defaultTLSSecret.name: <secret-name>
• Central redémarre automatiquement après modification de la CR
• Ce cert ne remplace que le TLS de l'UI/API — le mTLS interne (init bundle) est indépendant
• Vérification : openssl s_client -connect <host>:443 -servername <host>
• Après changement : vérifier l'état des Sensors dans Platform Configuration → Clusters
