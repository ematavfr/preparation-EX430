Points clés — Network Graph

  • 3 types d'entités : Internal (cluster) / External (hors cluster) / CIDR blocks
  • Le graphe ne se rafraîchit pas automatiquement — cliquer la notification en haut

Cycle de vie des Network Baselines

🔬
Phase Discovery
  • RHACS observe les connexions réseau du déploiement
  • Tous les flows observés sont automatiquement ajoutés à la baseline
  • Durée configurable (par défaut : quelques jours après déploiement)
  • Aucune alerte générée pendant cette phase
Nouveaux flows → baseline automatiquement
›
🔒
Baseline Établie
  • La baseline est figée — référence des connexions normales
  • Nouveaux flows = anomalous (hors baseline)
  • Toggle Enable alerts on baseline violations activé
  • Possibilité d'ajouter des flows à la baseline manuellement
  • Possibilité de marquer un flow baseline comme anomalous
Nouveaux flows → anomalous
›
🚨
Alertes Actives
  • Flow anomalous détecté → violation générée dans RHACS
  • Visible dans le menu Violations
  • Visible dans le Network Graph (indicateur visuel)
  • Peut déclencher des notifications (Slack, email…)
  • Action : Add to baseline ou Mark as anomalous
Alert on baseline violation = ON

Actions disponibles sur les flows (side panel → onglet Baseline)

Add to baseline
Ajoute un flow anomalous à la baseline de référence.
Ce flow ne déclenchera plus d'alerte.
Utile quand un nouveau service légitimes est déployé après la phase discovery.
Mark as anomalous
Retire un flow de la baseline — il devient anomalous.
Ce flow déclenchera désormais des alertes s'il est observé.
Utile pour nettoyer une baseline trop permissive.
Enable alerts on baseline violations
Toggle dans l'onglet Baseline du side panel.
Quand activé : tout flow hors baseline génère une violation RHACS.
À activer après la phase discovery quand la baseline est stable.
Export baseline as Network Policy
Génère un YAML NetworkPolicy Kubernetes basé sur les flows de la baseline.
Produit des règles ingress uniquement depuis l'UI.
À appliquer avec oc apply -f — RHACS ne peut pas appliquer directement.

Points clés — Baselines

  • Phase discovery : tous les flows sont ajoutés automatiquement — pas d'alertes
  • Après discovery : nouveaux flows = anomalous → alertes si toggle activé
  • Toggle "Enable alerts on baseline violations" = à activer manuellement
  • Add to baseline ↔ Mark as anomalous : actions réversibles
  • Export depuis la baseline → ingress uniquement (UI ne génère pas l'egress)
  • Chemin : Network Graph → clic déploiement → side panel → onglet Baseline

Génération de Network Policies

Via l'UI RHACS
Network → Network Graph → Network policy generator
Génère uniquement des règles ingress
Basé sur les flows observés dans le cluster
Bouton Generate and simulate → prévisualisation
Bouton Download YAML → export du fichier
Compare → diff côte à côte avec les policies existantes
Ne peut pas appliquer directement → oc apply -f requis
⚠️ Ingress uniquement — l'egress doit être géré séparément
Via roxctl (CLI)
roxctl netpol — analyse des manifests YAML
Génère ingress ET egress
Analyse les manifests locaux (pas besoin de Central connecté)
generate → génère les policies depuis les manifests
connectivity map → détecte les workloads non protégés
connectivity diff → compare deux versions de manifests
Exit code non-zéro si violation → intégration CI/CD
Clé pour l'examen : roxctl = ingress + egress (l'UI = ingress only)

Commandes roxctl netpol

# Générer des network policies depuis des manifests
roxctl netpol generate ./manifests/
# → Produit ingress + egress NetworkPolicy YAML

# Avec output dans un fichier
roxctl netpol generate ./manifests/ --output-file netpols.yaml
# Détecter les workloads non protégés (sans network policy)
roxctl netpol connectivity map ./manifests/
# → Affiche la matrice de connectivité inter-déploiements
# → Identifie les déploiements sans politique réseau
# Comparer la connectivité entre deux versions de manifests
roxctl netpol connectivity diff --dir1 ./v1/ --dir2 ./v2/
# → Affiche les changements de connectivité (ajouts/suppressions)
# → Exit code non-zéro si régression réseau détectée
# Télécharger et appliquer les policies générées par l'UI
# 1. UI : Network → Network policy generator → Generate → Download YAML
# 2. Appliquer manuellement :
oc apply -f generated-netpols.yaml
# L'UI RHACS ne peut pas appliquer directement les policies

Distinction critique pour l'examen

  • UI Network Policy Generator → génère ingress uniquement
  • roxctl netpol generate → génère ingress ET egress
  • L'UI ne peut jamais appliquer directement les policies → oc apply -f obligatoire
  • roxctl netpol connectivity map → détecter les workloads non protégés
  • roxctl netpol connectivity diff → comparer deux versions de manifests
  • Exit code non-zéro de roxctl → violation → pipeline CI/CD peut échouer (comportement voulu)
