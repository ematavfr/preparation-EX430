Troubleshooting — Registry

Tester depuis l'UI : Platform Configuration → Integrations → Image Integrations → [intégration] → Test
ErreurCause probableSolution
unauthorized: access to the requested resource is not authorized Credentials incorrects · Token expiré (ECR 12h) · Robot account sans droit pull Régénérer token/robot account · Vérifier permissions dans le registry · ECR : IAM role avec ecr:GetAuthorizationToken
x509: certificate signed by unknown authority Registry avec cert auto-signé ou CA interne non connu de RHACS Dev/test : décocher "Verify TLS" · Prod : ajouter le CA dans un Secret et le référencer dans la CR
dial tcp x.x.x.x:443: i/o timeout NetworkPolicy bloque Scanner → registry · Firewall · Registry interne non routable Vérifier NPs dans stackrox · Tester : oc exec -it scanner-xxx -- curl -v https://registry-url

Diagnostic réseau depuis le pod Scanner

# Tester la joignabilité du registry depuis le scanner
oc exec -it deployment/scanner -n stackrox -- \
  curl -v https://<registry-host>/v2/

# Vérifier les NetworkPolicies dans le namespace stackrox
oc get networkpolicies -n stackrox

# Logs scanner pour voir l'erreur exacte
oc logs -l app=scanner -n stackrox --tail=50

Troubleshooting — Sensor / Collector

Symptôme principal : cluster affiché Unhealthy (rouge) dans Platform Configuration → Clusters

Arbre de décision — Sensor unhealthy

Cluster Unhealthy dans l'UI
Vérifier les logs Sensor
oc logs -l app=sensor -n stackrox
certificate has expired
Init bundle expiré
Générer nouveau bundle + apply + restart sensor
connection refused / DNS failure
Central inaccessible
Vérifier centralEndpoint dans CR + DNS + route
secret not found
Secret init bundle supprimé
Re-appliquer le fichier YAML d'init bundle
ComposantProblèmeCauseSolution
Sensor Unhealthy / pas de connexion Init bundle expiré · réseau · secret corrompu Nouveau init bundle + oc rollout restart deployment/sensor
Collector Pod(s) en CrashLoopBackOff ou absent de certains nœuds Node tainté sans toleration · Kernel incompatible avec EBPF Ajouter taintToleration: TolerateTaints dans CR SecuredCluster · Passer en CORE_BPF
Admission Controller Workloads bloqués sans violation Timeout dépassé (fail-open) — Central lent ou injoignable Vérifier Central (resources/logs) · contactTimeout ajustable dans CR SecuredCluster 
# Renouveler l'init bundle
roxctl central init-bundles generate post-repair-bundle \
  --output-secrets new-bundle.yaml -e $ROX_ENDPOINT
oc apply -f new-bundle.yaml -n stackrox
oc rollout restart deployment/sensor -n stackrox

# Tester connectivité Sensor → Central
oc exec -it deployment/sensor -n stackrox -- \
  wget -qO- https://central.stackrox.svc:443/v1/ping

# Voir les tolerations du DaemonSet Collector
oc get ds collector -n stackrox -o yaml | grep -A 10 tolerations

Troubleshooting — OIDC

Config : Platform Configuration → Access Control → Auth Providers
ErreurCauseSolution
redirect_uri_mismatch URL de callback non déclarée dans le provider OIDC Ajouter dans le provider : https://<central-host>/sso/providers/oidc/callback
invalid_client client_id ou client_secret incorrect · client désactivé Vérifier/régénérer les credentials dans le provider OIDC · Réactiver le client
Login OK mais droits insuffisants (None) Claims du token ne correspondent pas aux mapping rules RHACS Décoder le JWT → comparer les claims → ajuster les rules dans Access Control
ERR_TOO_MANY_REDIRECTS Mode callback incorrect (fragment vs query) Dans l'auth provider RHACS, vérifier le "Callback Mode" (HTTP POST ou Fragment)

Arbre de décision OIDC

Problème de connexion OIDC
Erreur lors du login
redirect_uri / invalid_client
Vérifier config du client OIDC dans le provider
URL callback + client_id/secret
Login OK, rôle incorrect
None / droits insuffisants
Décoder JWT → aligner claims → mapping rules

Décoder un JWT pour diagnostiquer les claims

# Récupérer le token OIDC depuis les logs Central (mode debug)
oc logs -l app=central -n stackrox | grep -i "jwt\|claim\|token"

# Décoder manuellement (jwt.io ou en CLI)
echo "<token_base64>" | cut -d. -f2 | base64 -d | jq .

# URL callback attendue par RHACS
# https://<central-route>/sso/providers/oidc/callback
CENTRAL_HOST=$(oc get route central -n stackrox -o jsonpath='{.spec.host}')
echo "Callback URL : https://$CENTRAL_HOST/sso/providers/oidc/callback"

Troubleshooting — Object Storage S3 / ODF

Config : Platform Configuration → Integrations → Backup Integrations
ErreurCauseSolution
connection refused / i/o timeout Endpoint S3 incorrect · NetworkPolicy · TLS ODF auto-signé Vérifier l'URL endpoint · Tester curl depuis le pod central · Cocher "Disable TLS" pour ODF ou fournir le CA
AccessDenied Credentials incorrects · Bucket inexistant · Droits IAM insuffisants Vérifier Access Key/Secret Key · Créer le bucket · IAM : s3:PutObject, s3:GetObject, s3:ListBucket
SignatureDoesNotMatch ODF/Noobaa requiert Signature V2, RHACS/s3cmd envoie V4 par défaut s3cmd : signature_v2 = True dans ~/.s3cfg · RHACS : sélectionner endpoint "S3-compatible" non AWS
NoSuchBucket Nom de bucket incorrect ou bucket non créé Vérifier le nom exact : s3cmd ls · Créer le bucket si absent

Récupérer les infos bucket depuis OpenShift

# Lire le ConfigMap créé par ODF/Noobaa
oc extract --to=- cm/backup-acs -n integration-backup
# Retourne : BUCKET_NAME, BUCKET_HOST, BUCKET_PORT, BUCKET_REGION

# Récupérer le Secret avec les credentials
oc extract --to=- secret/backup-acs -n integration-backup

# Tester la connexion S3 depuis le pod Central
oc exec -it deployment/central -n stackrox -- \
  curl -v https://<s3-endpoint>/

# Lister les buckets avec s3cmd (debug)
s3cmd --config ~/.s3cfg ls --debug 2>&1 | grep -E "ERROR|bucket|endpoint"

Configuration s3cmd pour ODF/Noobaa

# ~/.s3cfg — valeurs clés pour ODF
[default]
access_key = <ACCESS_KEY>
secret_key = <SECRET_KEY>
host_base = s3-openshift-storage.apps.ocp4.example.com
host_bucket = s3-openshift-storage.apps.ocp4.example.com/%(bucket)s
signature_v2 = True    # OBLIGATOIRE pour ODF/Noobaa
use_https = True

Outils de diagnostic — Référence rapide

CommandeUsage
roxctl central debug dump -e $ROX_ENDPOINT Collecte tous les diagnostics (logs, config, state) dans un ZIP — premier réflexe Red Hat Support
oc logs -l app=central -n stackrox --tail=100 Logs Central (API, policy engine)
oc logs -l app=sensor -n stackrox --tail=100 Logs Sensor (connexion Central, policies)
oc logs -l app=collector -n stackrox Logs Collector (eBPF, événements réseau/processus)
oc logs -l app=scanner -n stackrox Logs Scanner (CVE, connexion registry)
oc get securedcluster -n stackrox -o yaml | grep -A 20 status État détaillé du SecuredCluster (conditions, erreurs)
oc exec -it deployment/sensor -n stackrox -- wget -qO- https://central.stackrox.svc:443/v1/ping Test connectivité Sensor → Central
curl -sk $ROX_ENDPOINT/v1/ping -H "Authorization: Bearer $TOKEN" Test API Central (vérifier que Central répond)

Priorité de diagnostic

1. Platform Configuration → System Health (vue globale)
2. Logs du composant incriminé (oc logs)
3. Test depuis l'UI (bouton Test des intégrations)
4. Vérifier credentials + réseau + certificats
5. roxctl central debug dump si escalade nécessaire
