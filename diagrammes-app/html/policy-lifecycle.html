Modes d'enforcement

Inform
Mode de détection — génère une violation sans bloquer
🔔
Violation créée dans le menu Violations de RHACS
📊
Visible dans le Dashboard et les rapports
✅
Le déploiement n'est pas bloqué — le pod tourne normalement
🔧
Applicable aux 3 stages : Build, Deploy, Runtime
💡
Mode par défaut pour les nouvelles policies et les policies système
Inform and Enforce
Mode actif — génère une violation ET applique une action corrective
Hard Enforcement (Deploy)
L'Admission Controller rejette la requête CREATE/UPDATE.
L'API server retourne une erreur 403 au client.
Le pod n'est jamais créé.

admission-control.bypass: Disabled requis dans SecuredCluster.
Soft Enforcement (Deploy)
Le pod est créé, puis RHACS le scale to 0.
Le Deployment reste à 0 replicas.
Utilisé quand l'Admission Controller n'est pas disponible
ou pour les ressources non gérées par le webhook.
Kill Pod (Runtime)
RHACS supprime le pod en cours d'exécution lors d'une violation runtime.
Si le Deployment a des replicas, OCP recrée le pod → nouvelle évaluation.
Utile pour stopper un processus malveillant détecté en cours d'exécution.

Points clés pour l'examen

  • Build → enforcement = Inform uniquement (pas de blocage possible au build)
  • Deploy Hard → bloque CREATE/UPDATE via webhook (Admission Controller requis)
  • Deploy Soft → scale to 0 après création (fallback ou absence de webhook)
  • Runtime → kill pod (le pod est supprimé lors de la violation)
  • Pour désactiver l'enforcement en urgence : oc delete validatingwebhookconfigurations

Actions sur les policies RHACS

✅
Enable / Disable
Active ou désactive une policy.
Une policy désactivée ne génère ni violation ni enforcement.
Toutes les policies système sont actives par défaut.
📋
Clone
Crée une copie modifiable d'une policy existante.
Indispensable pour modifier les policies système (non modifiables directement).
⚠ Toujours cloner avant de modifier une policy par défaut
✏️
Edit
Modifie les critères, la sévérité, les stages, l'enforcement ou les exclusions d'une policy.
Disponible uniquement sur les policies custom ou clonées.
🗑️
Delete
Supprime définitivement une policy custom.
Les policies système ne peuvent pas être supprimées — seulement désactivées.
📤
Export / Import
Exporte les policies au format JSON pour les partager ou les versionner.
Import possible depuis un fichier JSON via l'UI.
🎯
Scope (Exclusions)
Permet d'exclure des namespaces, clusters ou déploiements spécifiques de l'enforcement.
Utile pour les environnements de dev ou les outils d'infra (Prometheus, etc.).

Chemin dans l'UI

  • Platform Configuration → Policy Management — liste de toutes les policies
  • Filtrer par Policy categories, Severity, Lifecycle stage, Enforced
  • Menu ⋮ sur chaque policy → Enable / Disable / Clone / Delete / Export
  • Onglet Violations → voir toutes les violations actives par policy

Catégories de policies

Les catégories permettent de filtrer et organiser les policies dans l'UI. Une policy peut appartenir à plusieurs catégories.

Anomalous Activity
Processus inattendus, connexions anormales, escalade de privilèges
Cryptocurrency Mining
Détection de mineurs de crypto dans les containers
DevOps Best Practices
Bonnes pratiques CI/CD, Dockerfile, liveness probes…
Network Tools
curl, wget, netcat dans les images de prod
Package Management
apt, yum, pip exécutés au runtime
Privileges
Containers privilégiés, root, capabilities dangereuses
Security Best Practices
TLS, secrets en variables d'env, read-only filesystem…
Supply Chain Security
Images non signées, registries non autorisées
Vulnerability Management
CVE Critical/High, packages obsolètes
Docker CIS
Conformité CIS Docker Benchmark

Résumé pour l'examen

  • 3 stages : Build → Deploy → Runtime (chaque stage hérite du précédent)
  • 2 enforcement modes : Inform / Inform and Enforce
  • Deploy enforcement : Hard (block) vs Soft (scale to 0)
  • Runtime enforcement : Kill pod uniquement
  • Ne jamais modifier directement une policy système → toujours Clone d'abord
  • Policies système : Disable uniquement (pas de Delete)
  • Chemin UI : Platform Configuration → Policy Management
