Composants RHACS — Rôles détaillés

🏛️
Central
Central Cluster — Deployment
  • Console Web RHACS (port 443)
  • API REST et gRPC
  • Base de données PostgreSQL (violations, config, policies)
  • Point d'entrée unique pour administration
  • Reçoit les données de tous les Sensors
Namespace : stackrox
CR : Central
🔍
Scanner
Central Cluster — Deployment
  • Analyse les images pour détecter les CVE
  • Co-localisé avec Central dans le même namespace
  • Interroge les registries via les intégrations configurées
  • Résultats stockés dans la DB Central
  • Modes : Registry, Scanner, Registry+Scanner
Namespace : stackrox
CR : Central (spec.scanner)
📡
Sensor
Secured Cluster — Deployment
  • Agent principal du secured cluster
  • Surveille les ressources Kubernetes (pods, deployments, services…)
  • Communique avec Central en gRPC/TLS port 443
  • Coordonne Collector et Admission Controller
  • Reçoit les policies de Central et les applique localement
Namespace : stackrox
CR : SecuredCluster
🕵️
Collector
Secured Cluster — DaemonSet (1 pod / node)
  • Collecte les événements runtime sur chaque nœud
  • Surveille les connexions réseau (flux TCP/UDP)
  • Surveille les processus lancés dans les containers
  • Utilise eBPF ou un kernel module pour l'accès au syscall
  • Envoie les données au Sensor (communication locale)
Namespace : stackrox
DaemonSet — un pod par nœud
🛡️
Admission Controller
Secured Cluster — ValidatingWebhookConfiguration
  • Intercepte les requêtes CREATE/UPDATE vers l'API server
  • Évalue les policies RHACS en deploy-time
  • Hard enforcement : bloque la requête (reject 403)
  • Soft enforcement : autorise mais scale to 0 après déploiement
  • 3 replicas recommandés pour la HA
  • Bypass : bypass: Disabled dans SecuredCluster
Namespace : stackrox
CR : SecuredCluster (spec.admissionControl)
⚙️
RHACS Operator
Chaque cluster (Central + Secured)
  • Installé via OperatorHub sur chaque cluster
  • CR Central → déploie Central + Scanner
  • CR SecuredCluster → déploie Sensor + Collector + Admission Controller
  • Namespace : rhacs-operator (operator) / stackrox (composants)
  • Gère les mises à jour et la configuration RHACS
OperatorHub → RHACS Operator
Namespace : rhacs-operator

Flux de communication

Source Destination Protocole / Port Données transportées
Sensor Central gRPC / TLS 443 Alertes, événements Kubernetes, violations, état du cluster
Central Sensor gRPC / TLS 443 Policies, configuration, commandes de remédiation
Collector Sensor gRPC local Événements réseau, événements processus (syscalls)
Admission Controller Sensor HTTPS local AdmissionReview (CREATE/UPDATE) → décision allow/deny
API Server OCP Admission Controller HTTPS webhook Requêtes de création/modification de ressources
Scanner Image Registry HTTPS 443/5000 Pull des layers d'images pour analyse CVE
roxctl CLI Central API gRPC / TLS 443 image check, deployment check, backup, netpol…
Central S3 / GCS HTTPS 443 Backup automatique de la base de données PostgreSQL
Browser Central HTTPS 443 Console Web RHACS
Central OIDC Provider HTTPS 443 Authentification via Keycloak / OCP OAuth / Okta…

Note : Sensor initie la connexion vers Central

C'est toujours le Sensor qui établit la connexion vers Central (sortant du secured cluster vers le central cluster). Cela simplifie les règles firewall : seul le port 443 sortant depuis le secured cluster est requis. Central n'initie jamais de connexion vers les secured clusters.

Déploiement via l'Operator RHACS

ÉTAPE 1 Installer l'Operator sur chaque cluster
OperatorHub
OperatorHub → RHACS Operator
→ Install
Namespace : rhacs-operator
À faire sur le Central Cluster ET sur chaque Secured Cluster
Via CLI
oc apply -f rhacs-operator-subscription.yaml

Channel : stable
InstallPlan : Automatic
↓
ÉTAPE 2 Créer la CR Central (sur le Central Cluster)
CR Central — ce qu'elle déploie
→ Central (Deployment)
→ Scanner (Deployment + DB)
→ Route HTTPS vers Central
→ Secret central-tls (certificats)
→ Secret central-admin-password
Namespace
Créer d'abord : oc new-project stackrox

La CR Central se crée dans stackrox
Récupérer le mot de passe admin
oc -n stackrox get secret central-admin-password -o jsonpath='{.data.password}' | base64 -d
↓
ÉTAPE 3 Générer l'init bundle (connexion Sensor → Central)
Via l'UI RHACS
Platform Configuration
→ Clusters → New cluster
→ Generate bundle (Operator)
→ Télécharger le fichier YAML de secrets
Via roxctl
roxctl central init-bundles generate <nom> --output bundle.yaml

Appliquer sur le secured cluster :
oc apply -f bundle.yaml -n stackrox
Contenu du bundle
Secrets de certificats TLS permettant à Sensor d'authentifier Central

⚠️ À appliquer sur le secured cluster, pas sur Central
↓
ÉTAPE 4 Créer la CR SecuredCluster (sur chaque secured cluster)
CR SecuredCluster — ce qu'elle déploie
→ Sensor (Deployment)
→ Collector (DaemonSet — 1 pod/node)
→ Admission Controller (Deployment, 3 replicas)
→ ValidatingWebhookConfiguration
Spec clé
centralEndpoint: central.stackrox.svc:443
clusterName: mon-cluster
admissionControl.bypass: Disabled
admissionControl.replicas: 3
Namespace
Créer d'abord : oc new-project stackrox

La CR SecuredCluster se crée dans stackrox
