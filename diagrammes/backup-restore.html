Backup / Restore — RHACS

Types de backup · Intégration S3 · roxctl · s3cmd

🏠

2 types de backup dans RHACS

🗄️
Central Database Backup
Sauvegarde des données
Contenu : base PostgreSQL complète
Inclut : policies, violations, CVE, config, network baselines
Format : fichier .zip contenant la DB
Cas d'usage : récupération après corruption ou perte de données
Méthodes : automatique (S3) ou à la demande (roxctl)
Restore : roxctl central db restore <fichier.zip>
🚢
Central Deployment Backup
Sauvegarde de l'installation
Contenu : certificats TLS, secrets, manifests de déploiement
Inclut : init bundles, clés de chiffrement, configuration Operator
Cas d'usage : migration vers un autre cluster ou namespace
Ordre critique : restaurer les certificats d'abord
Pourquoi : les tokens Sensor et connections restent valides
Restore : via scripts RHACS ou l'Operator

Workflow complet — Backup DB puis Restore

1
Configurer les variables d'environnement
export ROX_ENDPOINT=central-stackrox.ocp4.example.com:443
export ROX_API_TOKEN=<api_token>
2
Déclencher le backup à la demande
roxctl central backup --output ~/backups/
→ Crée un fichier backup_YYYY-MM-DD_HH-MM.zip dans le répertoire spécifié
3
Restaurer la base de données
roxctl central db restore ~/backups/backup_2025-02-25.zip
→ Transfère le fichier vers Central et restaure la DB PostgreSQL
✓
Message de confirmation
Transfert : Transferring data at 1.9 MiB/s (ETA 00:00:00)
Succès : Successfully restored DB

Workflow migration — Restore du déploiement

1
Restaurer les certificats en premier
Restaurer les secrets TLS (init bundle, certificats Central)
oc apply -f certificates-bundle.yaml -n stackrox
⚠️ Obligatoire en premier — sinon les Sensors ne peuvent pas se reconnecter
2
Restaurer le déploiement Central
Via l'Operator : recréer la CR Central dans le namespace stackrox
Les Sensors se reconnectent automatiquement avec les certificats restaurés
3
Restaurer la base de données
roxctl central db restore <backup.zip>
Restaure policies, violations, CVE, config depuis le dernier backup DB

Points clés — types de backup

  • DB backup → roxctl central backup → fichier .zip
  • Deployment backup → certificats + secrets → migration cluster
  • Ordre migration : certificats d'abord → déploiement → DB restore
  • Message de succès restore : "Successfully restored DB"
  • Rôle requis pour backup/restore : Admin

Intégration Object Storage (backup automatique)

Paramètres de l'intégration S3
Integration name
Nom descriptif de l'intégration (ex: odf-integration)
Backups to retain
Nombre de fichiers à conserver (ex: 5)
Schedule interval
Fréquence : Daily ou Weekly
Bucket
Nom du bucket S3 (récupérer via cm/backup-acs)
Endpoint
URL endpoint S3 (route ODF ou URL AWS)
Region
us-east-1 pour ODF/Ceph
URL style
Path pour ODF (pas Virtual hosted)
Access Key ID
Clé d'accès S3 (récupérer via secret/backup-acs)
Secret Access Key
Clé secrète S3 (récupérer via secret/backup-acs)
Chemin UI + récupération credentials
Platform Configuration
→ Integrations
→ External Backups (ou Backup Integrations)
→ S3 API Compatible
→ New Integration
→ remplir les paramètres
→ Test (obligatoire)
→ Save
Trigger manuel :
menu ⋮ → Trigger backup
(pas de notification — vérifier avec s3cmd la)

Récupérer les credentials ODF (oc extract)

# Nom du bucket
oc extract --to=- cm/backup-acs -n integration-backup
# → BUCKET_NAME=backup-acs-b2cdf931-0ec5-41a9-9b31-033d360d3a38

# Credentials AWS
oc extract --to=- secret/backup-acs -n integration-backup
# → AWS_ACCESS_KEY_ID=...
# → AWS_SECRET_ACCESS_KEY=...

# Route publique S3 (endpoint)
oc get route s3 -n openshift-storage
# → s3-openshift-storage.apps.ocp4.example.com

Points clés — intégration S3

  • Chemin : Platform Configuration → Integrations → External Backups
  • Type : S3 API Compatible pour ODF/Ceph, MinIO (non officiel Red Hat)
  • URL style : Path pour ODF (pas Virtual hosted)
  • Toujours cliquer Test avant Save
  • Trigger manuel : menu ⋮ → Trigger backup (pas de notification — vérifier avec s3cmd la)
  • Config s3cmd : fichier ~/.s3cfg avec signature_v2 = True pour ODF

Commandes CLI — roxctl & s3cmd

roxctl — backup & restore

# Variables d'environnement (commun aux deux méthodes)
export ROX_ENDPOINT=central-stackrox.ocp4.example.com:443

# Méthode 1 : avec API token
export ROX_API_TOKEN=<api_token>
roxctl central backup

# Méthode 2 : avec mot de passe admin
roxctl -p <admin_password> central backup

# Avec répertoire de sortie
roxctl central backup --output /tmp/backups/
# → backup_2025-02-25_14-30.zip
# Restore de la base de données (rôle Admin requis)
export ROX_ENDPOINT=central-stackrox.ocp4.example.com:443

# Avec API token
export ROX_API_TOKEN=<api_token>
roxctl central db restore postgres_db.sql.zip

# Avec mot de passe admin
roxctl -p <admin_password> central db restore postgres_db.sql.zip
Transferring data at 1.9 MiB/s (ETA 00:00:00)
Successfully restored DB

s3cmd — vérification des backups dans le bucket

# Lister TOUS les objets de tous les buckets (commande principale de vérification)
s3cmd la
# 2025-02-25 12:19 9136871 s3://backup-acs-.../backup_2025-02-25.zip
# Lister les buckets disponibles
s3cmd ls

# Lister le contenu d'un bucket spécifique
s3cmd ls s3://backup-acs-b2cdf931-0ec5-41a9-9b31-033d360d3a38/

# Télécharger un fichier de backup
s3cmd get s3://bucket/acs/backup_2025-02-25.zip ./backup.zip

Authentification roxctl

Méthode Variable / Option Usage
API token export ROX_API_TOKEN=<token> Automatisation CI/CD — rôle Admin requis pour backup/restore
Mot de passe admin -p <admin_password> Utilisation manuelle / urgence
Endpoint export ROX_ENDPOINT=central:443 Commun aux deux méthodes — toujours requis

Résumé complet pour l'examen

  • 2 types : DB backup (données) vs Deployment backup (config/migration)
  • Backup CLI : roxctl central backup → fichier .zip
  • Restore CLI : roxctl central db restore <fichier.zip>
  • Message succès : "Successfully restored DB"
  • Migration : certificats en premier → déploiement → DB restore
  • Backup auto S3 : Platform Config → Integrations → External Backups
  • Vérification bucket : s3cmd la (liste tous les objets)
  • s3cmd config : ~/.s3cfg — signature_v2 = True requis pour ODF
  • Rôle requis : Admin pour toutes les opérations backup/restore
