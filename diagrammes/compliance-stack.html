Stack Compliance — de l'Operator au scan

⚠️ Le Compliance Operator doit être installé sur le cluster Central ET sur chaque cluster managé
⚙️
Compliance Operator
OperatorHub → openshift-compliance
  • Installé depuis OperatorHub sur chaque cluster
  • Namespace : openshift-compliance
  • Fournit les CRD : ScanSetting, ScanSettingBinding, TailoredProfile, ComplianceScan…
  • Exécute les scans via des pods sur les nœuds (master et worker)
Namespace : openshift-compliance
configure
⏱️
ScanSetting
kind: ScanSetting
  • Définit quand et comment scanner
  • schedule — cron expression (ex: 0 1 * * *)
  • roles — nœuds ciblés : master, worker
  • rawResultStorage.size — taille de stockage des résultats
  • autoApplyRemediations — appliquer les remédiation auto
Namespace : openshift-compliance
référencé par
🔗
ScanSettingBinding
kind: ScanSettingBinding
  • Lie un ScanSetting à un ou plusieurs profils
  • Champ settingsRef → nom du ScanSetting
  • Champ profiles → liste des profils (Profile ou TailoredProfile)
  • Déclenchement via : oc compliance bind
  • Crée un ComplianceScan pour chaque profil
Namespace : openshift-compliance
pointe vers
📋
Profile
kind: Profile
  • Profil de conformité standard (fourni par l'Operator)
  • Ex : ocp4-cis, ocp4-pci-dss
  • Non modifiable directement
  • Référencé par kind: Profile dans le binding
✂️
TailoredProfile
kind: TailoredProfile
  • Profil personnalisé basé sur un profil existant
  • extends → profil source
  • disableRules / enableRules
  • setValues → modifier les variables
  • Référencé par kind: TailoredProfile dans le binding

Points clés — stack

  • Ordre : Operator → ScanSetting → ScanSettingBinding → Profile/TailoredProfile
  • Compliance Operator sur Central ET managed clusters
  • Namespace unique : openshift-compliance
  • Déclencher un scan : oc compliance bind ou créer un ScanSettingBinding
  • Dans le binding : Profile → kind: Profile, TailoredProfile → kind: TailoredProfile

YAML des ressources Compliance

ScanSettingBinding
TailoredProfile
apiVersion: compliance.openshift.io/v1alpha1
kind: ScanSetting
metadata:
  name: my-scan-setting
  namespace: openshift-compliance
spec:
  schedule: "0 1 * * *"  # Tous les jours à 1h UTC
  roles:
    # Nœuds ciblés par le scan
    - master
    - worker
  rawResultStorage:
    size: "1Gi"
    rotation: 3       # Nombre de scans conservés
  autoApplyRemediations: false # Jamais en prod sans test !
  autoUpdateRemediations: false
apiVersion: compliance.openshift.io/v1alpha1
kind: ScanSettingBinding
metadata:
  name: my-binding
  namespace: openshift-compliance
spec:
  settingsRef:
    name: my-scan-setting # Référence au ScanSetting
    kind: ScanSetting
  profiles:
    # Profil standard
    - name: ocp4-cis
      kind: Profile           # ← kind: Profile
      apiGroup: compliance.openshift.io/v1alpha1
    # Profil personnalisé
    - name: my-tailored-profile
      kind: TailoredProfile   # ← kind: TailoredProfile !
      apiGroup: compliance.openshift.io/v1alpha1
apiVersion: compliance.openshift.io/v1alpha1
kind: TailoredProfile
metadata:
  name: my-tailored-cis
  namespace: openshift-compliance
spec:
  extends: ocp4-cis          # Profil source
  description: "CIS customisé"
  disableRules:
    - name: ocp4-api-server-audit-log-maxage
      rationale: "Géré par la politique interne"
  enableRules:
    - name: ocp4-scc-limit-privileged-containers
      rationale: "Requis par politique sécurité"
  setValues:
    - name: ocp4-var-role-master
      rationale: "Adapter au rôle de nœud"
      value: "master"
⚠️ TailoredProfile ne peut qu'activer/désactiver/modifier des règles existantes — il ne peut pas créer de nouvelles règles.

Points clés — YAML

  • ScanSetting définit le schedule cron et les roles (master/worker)
  • ScanSettingBinding → settingsRef + profiles (liste mixte Profile/TailoredProfile)
  • Profile standard → kind: Profile | Profil custom → kind: TailoredProfile
  • autoApplyRemediations: false recommandé en production
  • TailoredProfile → extends obligatoire — ne peut pas créer de règles ex-nihilo
  • Lister les variables disponibles : oc get variables -n openshift-compliance

Distinction Control vs Check

Control
Exigence de haut niveau dans un standard de conformité.

Ex : "Ensure audit logging is enabled"

⚠️ Un seul Check échoué dans un Control = le Control entier échoue.
Check
Test technique individuel vérifiant un aspect précis d'un Control.

Ex : "Verify --audit-log-maxage is set to 30 or as appropriate"

Un Control peut avoir plusieurs Checks — tous doivent passer pour que le Control soit PASS.

Points clés — Dashboard

  • 5 standards natifs : CIS K8s, HIPAA, NIST 800-190, NIST 800-53 Rev.4, PCI DSS 3.2.1
  • Control ≠ Check : un Check failed = Control entier failed
  • Page Coverage nécessite le Compliance Operator installé sur le cluster
  • Export rapports : PDF (résumé exécutif) ou CSV zippé (preuves détaillées)
  • Chemin : Compliance → Compliance Management
  • Scan manuel : bouton Scan Environment dans le dashboard

TailoredProfile — Personnalisation des profils

extends
Champ obligatoire — indique le profil source à personnaliser.

Exemples : ocp4-cis, ocp4-pci-dss, ocp4-nist-800-53-rev4

Un TailoredProfile ne peut pas être créé ex-nihilo — il doit toujours étendre un profil existant.
⚠️ Impossible de créer des règles — seulement activer/désactiver/modifier
disableRules
Désactive des règles du profil source.

Format : name (nom de la règle) + rationale (justification obligatoire)

Cas d'usage : règle non applicable à l'environnement, gérée par un autre outil, exception documentée.
enableRules
Active des règles qui sont désactivées dans le profil source.

Même format : name + rationale

Cas d'usage : durcir un profil existant en activant des règles supplémentaires.
setValues
Modifie la valeur d'une variable de règle.

Format : name (variable) + value + rationale

Lister les variables disponibles :
oc get variables -n openshift-compliance

Workflow TailoredProfile → ScanSettingBinding

# 1. Lister les profils disponibles
oc get profiles.compliance -n openshift-compliance

# 2. Lister les variables modifiables
oc get variables -n openshift-compliance

# 3. Créer le TailoredProfile
oc apply -f tailored-profile.yaml

# 4. Vérifier le statut du TailoredProfile
oc get tailoredprofile -n openshift-compliance

# 5. Lier via oc compliance bind (alternative au YAML ScanSettingBinding)
oc compliance bind -N my-binding tailoredprofile/my-tailored-cis

# 6. Suivre l'état du scan
oc get compliancescans -n openshift-compliance

Points clés — TailoredProfile

  • extends est obligatoire — un TP étend toujours un profil existant
  • Impossible de créer de nouvelles règles — seulement activer/désactiver/modifier
  • rationale est requis pour chaque règle activée/désactivée — justification documentée
  • Dans ScanSettingBinding → kind: TailoredProfile (pas kind: Profile)
  • Commande bind : oc compliance bind -N <nom> tailoredprofile/<nom>
  • Variables disponibles : oc get variables -n openshift-compliance
  • Profils disponibles : oc get profiles.compliance -n openshift-compliance
