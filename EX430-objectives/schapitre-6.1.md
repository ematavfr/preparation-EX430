# 6.1 — Integrate external private image registries

## Pourquoi intégrer un registry externe ?

RHACS a besoin des credentials du registry pour :
1. **Scanner les images** (pull indirect via Scanner)
2. **Enrichir les métadonnées** (manifest, layers)
3. **Déclencher des scans automatiques** à la notification d'un push

## Types d'intégrations supportées

| Type | Description |
|---|---|
| **Docker Hub** | Registry public Docker |
| **Quay.io / Red Hat Quay** | Registry Red Hat (cloud et on-prem) |
| **ECR** (AWS) | Elastic Container Registry |
| **GCR / GAR** (Google) | Google Container Registry / Artifact Registry |
| **ACR** (Azure) | Azure Container Registry |
| **JFrog Artifactory** | Registry d'entreprise |
| **IBM Cloud Container Registry** | IBM Cloud |
| **Generic Docker** | Tout registry compatible Docker v2 API |
| **Nexus** | Sonatype Nexus |
| **GitHub GHCR** | GitHub Container Registry |

## Configurer une intégration registry

### Via l'UI

1. **Platform Configuration → Integrations → Image Integrations**
2. Sélectionner le type (ex: `Generic Docker`)
3. Renseigner :

```
Name: quay-production
Endpoint: quay.io (ou URL interne)
Username: robot-account
Password / Token: ****
TLS: Verify TLS (décocher uniquement si registry interne auto-signé)
Categories: Registry + Scanner (si le registry peut scanner)
```

4. **Test Integration → Save**

### Via l'API

```bash
curl -sk -X POST "$ROX_ENDPOINT/v1/imageintegrations" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "quay-production",
    "type": "quay",
    "categories": ["REGISTRY"],
    "quay": {
      "endpoint": "quay.io",
      "oauthToken": "YOUR_ROBOT_TOKEN"
    },
    "autogenerated": false,
    "skipTestIntegration": false
  }'
```

## Cas spéciaux

### Quay — 3 méthodes d'intégration

| Méthode | Authentification | Quand l'utiliser |
|---|---|---|
| **Public** | Aucune | Registry public Quay.io |
| **Robot account** | Token robot account | **Recommandé** — registry privé Quay |
| **OAuth token** | OAuth API token | Utiliser le **scanner Quay** à la place du scanner RHACS |

> Le robot account est la méthode **recommandée** pour les registries privés Quay. L'OAuth token n'est nécessaire que si on veut déléguer le scan à Quay lui-même.

### ECR (AWS)

```json
{
  "type": "ecr",
  "ecr": {
    "registryId": "123456789012",
    "region": "eu-west-1",
    "useIam": true
  }
}
```

> `useIam: true` utilise l'IAM role attaché au pod Scanner (IRSA sur EKS).

### Registry avec TLS auto-signé

```bash
# Dans l'UI : décocher "Verify TLS"
# Ou via CR SecuredCluster pour le scanner local :
spec:
  scanner:
    customTLSCertificate:
      name: my-registry-ca
      namespace: stackrox
```

### OpenShift Internal Registry

```bash
# Récupérer le token SA avec accès image-puller
TOKEN=$(oc sa get-token rhacs-scanner -n stackrox)
# Endpoint : image-registry.openshift-image-registry.svc:5000
```

## Vérifier que le scan fonctionne

```bash
# Forcer le scan d'une image
roxctl image scan --image=quay.io/myorg/myapp:1.0 \
  -e $ROX_ENDPOINT

# Vérifier dans l'UI : Vulnerability Management → Images → chercher l'image
```

---

## Résumé pour l'examen

> - Intégrations registry : **Platform Configuration → Integrations → Image Integrations**
> - Types clés : Quay, ECR (IAM), GCR, ACR, Generic Docker
> - **Tester l'intégration** avant de sauvegarder
> - Categories : `REGISTRY` (scan) et/ou `SCANNER` (si le registry peut scanner lui-même)
> - Registry TLS auto-signé : décocher "Verify TLS" ou fournir le CA
> - Robot accounts recommandés (permissions minimales : pull seulement)
